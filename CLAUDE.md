# Calmdromeda — Claude 작업 규칙

## 커뮤니케이션 순서 (필수 준수)

```
[요건검토] 발생 케이스 나열 + 세부 로직 방향 합의
    ↓ (합의 완료)
[계획] 파일명:라인 기준 ASIS/TOBE
    ↓ (컨펌)
[코딩] 전체 변경 한 번에
    ↓
[검증] Claude 먼저 단위테스트 → 결과 공유
    ↓ (이상 없으면)
[사용자 로컬 테스트 → 직접 커밋]
```

---

## 규칙 1: 요건검토 — 케이스 나열 + 세부 로직 합의

- 계획 단계 진입 전에 반드시 요건검토를 먼저 진행한다.
- 발생할 수 있는 케이스들을 제안하며 세부 로직 방향을 함께 검토한다.
- Claude와 사용자가 동일한 로직을 얘기하고 있는지 세부 내용까지 확인 후 진행.

## 규칙 2: 계획 — 파일명:라인 기준 ASIS/TOBE

- 계획은 항상 `파일명:라인번호` 기준으로 작성한다.
- 사용자가 열어서 해당 라인을 바로 찾을 수 있게 구체적으로 명시.
- 여러 파일에 걸친 변경이면 전체 ASIS/TOBE를 한 번에 보여준다.

## 규칙 3: 코딩 — 컨펌 후 한 번에

- 코드 수정은 사용자 컨펌 1번 이후 전체를 한 번에 수정한다.
- 파일별로 쪼개서 중간중간 수정하지 않는다.
- 코드 파악(읽기)은 컨펌 없이 진행 가능.

## 규칙 4: 검증 — Claude 단위테스트 먼저

- 코딩 완료 후 Claude가 먼저 논의한 로직대로 동작하는지, 오류는 없는지 단위테스트 수행.
- 결과를 사용자에게 공유한 뒤 사용자가 로컬 테스트 진행.
- 검증 전 커밋하지 않는다.

## 규칙 5: 커밋/푸시/PR

- 커밋/푸시는 사용자가 로컬 테스트까지 마친 후 **명시적 컨펌**을 줬을 때만 진행.
- PR은 작업이 완전히 마무리됐을 때 사용자가 명시적으로 요청할 때만 생성한다.
- PR 머지는 사용자가 GitHub에서 직접 검토 후 진행.

## 규칙 6: 세션 논의가 길어질 때

- 논의 중에는 코드를 건드리지 않고 대화로만 진행한다.
- 최종 합의된 변경사항을 모아서 한 번에 수정한다.
